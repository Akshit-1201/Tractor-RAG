# Tractor Maintenance Assistant

A production-shaped **RAG (Retrieval-Augmented Generation) web application** for a tractor
manufacturer. Administrators upload the official reference material — PDF manuals, parts
catalogs, and reference images of parts, engine layouts, and dashboard warning lights.
Customers ask troubleshooting questions in plain language and receive answers grounded
**strictly** in that material, with the relevant reference image displayed when — and only
when — it was actually used to produce the answer. When the answer isn't in the uploaded
content, the assistant says so explicitly. It never guesses.

| Surface | URL | Access |
|---|---|---|
| 💬 Customer chat | `http://localhost:5173/` | Public, text-only, no login |
| 🔐 Admin dashboard | "Login as Admin" button on the chat page (or `http://localhost:5173/admin/login`) | JWT login (seeded admin account) |
| ⚙️ API docs (Swagger) | `http://localhost:8000/docs` | Auto-generated by FastAPI |

## 1. Features

**Admin (authenticated)**

- Log in to a JWT-guarded dashboard (single seeded admin; password managed via `.env`).
- Upload PDF documents (manuals, parts catalogs, maintenance guides) by drag-and-drop.
- Upload reference images (warning lights, parts diagrams, engine layouts).
- Watch ingestion status flip live: `PROCESSING → INDEXED` (or `FAILED`).
- Delete any document or image — its indexed chunks and stored file are removed with it.
- Analytics dashboard: total questions, answered vs. "don't know" rate, most common
  topics, and a live recent-questions feed.

**Customer (public)**

- Ask questions in natural language; answers **stream in token by token** (SSE).
- Every answer is grounded only in admin-uploaded content and lists its **sources**.
- When a reference image was actually used to answer, it appears **beneath the answer**.
- Out-of-scope questions get the explicit refusal:
  *"I don't have information about that in the available documents."* — never a guess.
- Exact part numbers and error codes (e.g. `E-047`, `AL-120`) resolve via lexical search.

## 2. Architecture

See the annotated diagram: **[docs/architecture.svg](docs/architecture.svg)**

The system is two distinct paths, and every design decision follows from keeping them apart:

```
UPLOAD / INGESTION PATH (runs rarely — all expensive work lives here)
  PDF   → parse (PyMuPDF) → chunk (~650 tokens, paragraph boundaries, overlap)
        → embed → index
  Image → ONE vision call (gpt-4o-mini) → rich text description
        → embed → index into the SAME table

QUERY / HOT PATH (runs on every question — deliberately cheap)
  question → embed → hybrid retrieval (pgvector cosine + Postgres full-text,
  fused with Reciprocal Rank Fusion) → one grounded gpt-4o-mini call
  → parse citations → three-gate image check → stream via SSE
  → log to analytics.  NO vision. NO reranker. Ever.
```

**Backend layering:** thin FastAPI routers (validation only) → service layer
(`ingestion · vision · embeddings · retrieval · chat · analytics`) → a single
**PostgreSQL + pgvector** store holding vectors, relational data, the lexical index,
and analytics — one database instead of three services.

**Data model (5 tables):** `admins` (credentials) · `documents` / `images` (upload
metadata + status) · `chunks` (the **unified index** — a document text chunk and an
image description are the same kind of row, distinguished by `source_type`) ·
`questions` (one analytics row per customer question).

## 3. Key design decisions

**The unified chunk index.** Image descriptions and document chunks live in one table
and are retrieved identically. Deleting a document or image cascade-deletes its chunks,
so the index always mirrors exactly what the admin has uploaded — which keeps the
grounding guarantee honest.

**Vision at upload, never per query.** Each image is described by a vision model
exactly once, at ingestion. Cost scales with uploads (rare), not traffic (constant).
When descriptions proved lossy during tuning (the model asserted "not blinking" for a
still image), the fix was a richer *ingestion* prompt — paid once — not query-time
vision.

**Hybrid retrieval with honest score semantics.** Dense (pgvector cosine) and lexical
(Postgres full-text) results are fused by Reciprocal Rank Fusion — but RRF only
*orders* the list. Every chunk keeps its **dense cosine similarity** as its score,
because the image gate's threshold compares against it (RRF scores top out near 0.03
and would disable images entirely).

**The image gate (Option A).** An image is shown only if **all three** hold:
(1) the question was actually answered, (2) the LLM **cited** the image's chunk —
retrieved ≠ used, (3) its cosine similarity clears `IMAGE_SIMILARITY_THRESHOLD`.
Verified live: a flashing-battery-light question shows the battery image, while an
engine-oil question that *retrieves* the engine diagram (but doesn't use it) shows
nothing. When in doubt, no image — a text answer is never wrong.

**The two-layer "I don't know" guarantee.** The prompt demands the exact refusal
string; the service layer independently detects refusals (normalized full-phrase
matching that survives curly apostrophes, politeness prefixes, and blank outputs),
canonicalizes the reply, and force-drops any citations — so an unanswered question can
never carry sources or an image.

**SSE with a clean stream.** Answers stream token by token; the model's trailing
`CITED: [...]` bookkeeping line is stripped server-side with a hold-back buffer (it
never reaches the customer, even split across token boundaries), and the final SSE
event carries the authoritative answer, sources, and image.

## 4. Tech stack

| Layer | Choice |
|---|---|
| Backend | Python 3.12 · FastAPI · SQLAlchemy 2 · Alembic migrations |
| Store | PostgreSQL 16 + pgvector (HNSW index) — vectors, relational data, full-text, analytics |
| AI | OpenAI `text-embedding-3-small` (1536-dim) · `gpt-4o-mini` (chat + vision) |
| Frontend | React 18 + Vite 6, TypeScript, React Router |
| Auth | JWT (python-jose) + bcrypt (passlib) |
| Packaging | Docker Compose (3 services: `db`, `backend`, `frontend`) |

All model names, thresholds, and limits live in `.env` — nothing is hard-coded.

## 5. Prerequisites

- **Docker Desktop**, installed and running (WSL2 backend on Windows). Verify with
  `docker --version` and `docker info`.
- An **OpenAI API key** from <https://platform.openai.com/api-keys>. Used for
  embeddings, chat, and per-image vision calls — a complete demo costs well under $0.10.
- ~2 GB of free disk for images and volumes.

No local Python or Node installation is needed — everything runs in containers.

## 6. Running the project — step by step

### Step 1 — Get the code

```bash
git clone <repository-url>
cd Tractor-RAG
```

### Step 2 — Configure your environment

```bash
cp .env.example .env
```

Open `.env` and set three values (everything else has working defaults):

| Variable | What to put there |
|---|---|
| `OPENAI_API_KEY` | Your real OpenAI key (`sk-...`). Required for ingestion and chat. |
| `ADMIN_PASSWORD` | The admin login password. The account is created from this on first boot and **re-synced from it on every restart** — changing it here changes the login. |
| `JWT_SECRET` | Any long random string (session token signing key). |

### Step 3 — Build and start everything

```bash
docker compose up --build
```

First run takes a few minutes (image pulls + dependency installs). What happens on boot,
in order:

1. Postgres starts; a healthcheck gates the backend until the database accepts connections.
2. The backend runs Alembic migration `0001`: enables the `vector` extension and creates
   all five tables, indexes, and the full-text trigger.
3. `scripts/seed_admin.py` creates (or re-syncs) the admin account from `.env`.
4. The API starts on port `8000`; the frontend dev server starts on port `5173`.

### Step 4 — Verify it's up

```bash
curl http://localhost:8000/api/health
# → {"status":"ok"}
```

Then open:

- **Customer chat:** <http://localhost:5173/>
- **Admin dashboard:** click **"Login as Admin"** in the chat page header (or go to
  <http://localhost:5173/admin/login> directly) — sign in with
  `ADMIN_USERNAME` / `ADMIN_PASSWORD` from your `.env`; you land on the content page.

### Step 5 — Load content

The chat can only answer from uploaded content, so load some. Easiest — the bundled
demo corpus (an operator manual, a parts & maintenance guide, a flashing battery
warning light, and an engine layout diagram):

```bash
docker compose exec backend python -m scripts.seed_content
```

The script uploads through the real admin API and waits for indexing; it is
**idempotent** (re-running skips anything already present). Alternatively, drag and
drop your own PDFs/images at `/admin/content` and watch the status flip
`PROCESSING → INDEXED`.

### Step 6 — Ask questions

At <http://localhost:5173/>, try these (with the demo corpus loaded):

| Question | Expected result |
|---|---|
| *What does a flashing red battery light mean?* | Grounded answer **+ the warning-light image beneath it + sources** |
| *How do I change the transmission fluid?* | Step-by-step answer with sources, no image |
| *How do I change the engine oil?* | Answer from the manuals; the engine diagram is retrieved but **not shown** (the model didn't use it — retrieved ≠ used) |
| *What does error code E-047 mean?* | Exact error-code answer, rescued by lexical search |
| *What's the best fertilizer for wheat?* | Exactly: *"I don't have information about that in the available documents."* — no sources, no image |

Then open **Admin → Analytics** to see every question logged: totals, answer rate, top
topics, and the recent-questions feed.

> Tip: the very first question after boot is slower (cold OpenAI connection). Ask a
> throwaway question first if you're demoing.

## 7. Testing the project — step by step

### Step 1 — Run the full backend suite

The stack must be up (`docker compose up`) because DB-backed tests run against the
compose database:

```bash
docker compose exec -T backend pytest -q
```

Expected output: **`58 passed`** (a handful of third-party deprecation warnings from
PyMuPDF/passlib/jose are normal). The suite creates and cleans its own fixtures — it
passes against a completely empty database, and it never calls OpenAI (all AI calls are
mocked with deterministic stand-ins).

### Step 2 — Know what the tests prove

The suite is weighted toward the **accuracy criterion** — the properties this project
is scored on:

| Area | What is proven |
|---|---|
| **Grounding refusal** | Out-of-scope question → the exact IDK string, `is_answered=false`, no sources, no image — even when the model emits IDK *plus* citations, or an empty/prefixed refusal |
| **Image gate — all 4 cases** | happy path (cited + above threshold → shown) · retrieved-but-not-cited → hidden · no image on IDK · cited but below threshold → hidden |
| **Retrieval** | RRF ranks two-signal agreement on top; an exact code match (`E-047`) found only by lexical search is rescued into the top-k; fusion never overwrites the dense cosine scores the gate depends on |
| **SSE integrity** | Token events then a final metadata event; the `CITED:` trailer never reaches the client, even split mid-word (`CIT` + `ED:`) or as the entire output |
| **Citations** | `CITED: [1, 3]` parsing, out-of-range numbers dropped, missing trailer treated conservatively |
| **Auth** | Login round-trip, wrong password → 401, admin routes reject missing/garbage tokens, login brute-force → 429 |
| **Ingestion** | PDF → indexed with embeddings + populated `tsv`; image → exactly one chunk with `image_url` metadata; failures mark `FAILED`; deletes cascade to chunks and files |
| **Upload validation** | Wrong types → 415; oversized → 413 at both layers (Content-Length precheck + streaming cap with partial-file cleanup) |
| **Rate limiting** | 21st chat request in a minute → 429; over-length question → 422 before any LLM call |
| **Analytics** | Endpoint shape per spec, guarded, correct computation, zero-safe on an empty table |
| **Infra** | Config loads from env, health endpoint, migration creates schema + extension, admin seed is an idempotent upsert |

### Step 3 — Useful test selections

```bash
docker compose exec -T backend pytest tests/test_chat_service.py -q   # image gate + IDK logic
docker compose exec -T backend pytest tests/test_chat_api.py -q       # endpoint + SSE behavior
docker compose exec -T backend pytest -k "image_gate" -q              # just the four gate cases
docker compose exec -T backend pytest -q -v                           # verbose test names
```

### Step 4 — Frontend type-check

```bash
docker compose exec -T frontend npx tsc --noEmit
```

Expected: no output, exit code 0.

### Step 5 — Manual end-to-end checklist

With seed content loaded, walk the running app once:

1. Visit `/admin/content` **without logging in** → redirected to `/admin/login`. ✔ role separation
2. Log in → upload any small PDF → status flips `PROCESSING → INDEXED` → delete it. ✔ ingestion + cascade
3. Ask the five questions from [Step 6 above](#step-6--ask-questions) and confirm each
   expected result — streaming text, image on the battery question only, sources under
   answered replies, the plain refusal for fertilizer. ✔ accuracy + image gate
4. Open **Analytics** → the questions you just asked appear, with the refusal counted
   as unanswered. ✔ analytics loop
5. `curl http://localhost:8000/api/admin/documents` → `401`. ✔ API guard

## 8. API reference

Interactive Swagger docs: `http://localhost:8000/docs`

| Method & path | Auth | Description |
|---|---|---|
| `GET /api/health` | — | Liveness check → `{"status":"ok"}` |
| `POST /api/admin/login` | — (rate-limited 10/min) | `{username, password}` → `{access_token, token_type}` |
| `POST /api/admin/documents` | Bearer JWT | Multipart PDF upload → `202 {id, filename, status}`; ingestion runs in the background |
| `GET /api/admin/documents` | Bearer JWT | List with `status` and `chunk_count` |
| `DELETE /api/admin/documents/{id}` | Bearer JWT | `204` — removes row, file, and all its chunks |
| `POST /api/admin/images` | Bearer JWT | Multipart image upload (PNG/JPEG/WebP) → `202`; one vision call at ingestion |
| `GET /api/admin/images` | Bearer JWT | List with category, description, `image_url` |
| `DELETE /api/admin/images/{id}` | Bearer JWT | `204` — removes row, file, and its chunk |
| `GET /api/admin/analytics` | Bearer JWT | Totals, answered/unknown split, answer rate, top topics, recent questions |
| `POST /api/chat` | — (rate-limited 20/min) | `{question}` → JSON answer, or SSE stream with `Accept: text/event-stream` |
| `GET /storage/images/{file}` | — | Public reference images (raw PDFs are deliberately **not** served) |

**Chat, JSON mode:**

```bash
curl -X POST http://localhost:8000/api/chat \
  -H "Content-Type: application/json" \
  -d '{"question": "What does a flashing red battery light mean?"}'
```

```json
{
  "answer": "A FLASHING red battery light indicates a charging system fault...",
  "is_answered": true,
  "sources": [
    { "type": "document", "name": "demo_tractor_manual.pdf", "chunk_id": 42 },
    { "type": "image", "name": "flashing_battery_warning_light.png", "chunk_id": 88 }
  ],
  "image": { "url": "/storage/images/...png", "caption": "flashing_battery_warning_light.png" }
}
```

**Chat, SSE mode** (add `Accept: text/event-stream`): a series of
`event: token / data: {"text": "..."}` events, then one
`event: final / data: {answer, is_answered, sources, image}` carrying the authoritative
result.

## 9. Configuration reference

All configuration is environment-driven (`.env`, documented in `.env.example`):

| Variable | Default | Purpose |
|---|---|---|
| `OPENAI_API_KEY` | — | **Required.** Embeddings, chat, and vision calls |
| `EMBEDDING_MODEL` | `text-embedding-3-small` | 1536-dim embedding model (both paths) |
| `CHAT_MODEL` / `VISION_MODEL` | `gpt-4o-mini` | Grounded answering / image description |
| `DATABASE_URL` | compose-internal Postgres | SQLAlchemy connection string |
| `JWT_SECRET` / `JWT_EXPIRY_MINUTES` | `change-me` / `120` | Token signing + expiry |
| `ADMIN_USERNAME` / `ADMIN_PASSWORD` | `admin` / `change-me` | Seeded admin credential (re-synced each boot) |
| `RETRIEVAL_TOP_K` | `8` | Chunks returned by hybrid retrieval |
| `IMAGE_SIMILARITY_THRESHOLD` | `0.35` | Image gate 3 — compared against **dense cosine similarity** |
| `CHUNK_TARGET_TOKENS` / `CHUNK_OVERLAP_TOKENS` | `650` / `80` | Chunking geometry |
| `MAX_QUESTION_CHARS` | `1000` | Chat questions longer than this → 422 (also caps the UI input) |
| `CHAT_RATE_LIMIT` | `20/minute` | Per-IP limit on the public chat |
| `LOGIN_RATE_LIMIT` | `10/minute` | Per-IP brute-force guard on login |
| `MAX_UPLOAD_MB` | `50` | Per-file upload cap (early Content-Length reject + exact stream cap) |
| `STORAGE_PATH` | `/data/storage` | Upload storage inside the backend container |
| `FRONTEND_ORIGIN` | `http://localhost:5173` | CORS allow-list |

The RAG tuning values were verified live against the four worked examples in the design
spec (Appendix A) using the bundled seed corpus.

## 11. Development mode

`docker-compose.override.yml` is applied automatically by `docker compose up` and
bind-mounts the source into both containers:

- Backend edits hot-reload via `uvicorn --reload`.
- Frontend edits hot-reload via Vite HMR.
- Tests always run against your current working tree — no rebuilds needed.

Delete or rename the override file to run the plain production-shaped configuration.
After changing `.env`, recreate the backend so it sees new values:
`docker compose up -d --force-recreate backend`.

## 12. Troubleshooting

| Symptom | Cause / fix |
|---|---|
| `error during connect: ... dockerDesktopLinuxEngine` | Docker Desktop isn't running — start it, retry |
| Port `8000` / `5173` already in use | Stop the conflicting process or change the mapping in `docker-compose.yml` |
| Upload lands on `FAILED` | Almost always a missing/invalid `OPENAI_API_KEY`. Check `docker compose logs backend`, fix `.env`, `docker compose up -d --force-recreate backend`, delete and re-upload |
| Every question gets "I don't have information…" | Nothing indexed yet — run the seed script (Step 5) or upload content |
| `429 Too Many Requests` | Per-IP rate limits (`CHAT_RATE_LIMIT`, `LOGIN_RATE_LIMIT`) — wait a minute |
| First answer after boot is slow | Cold OpenAI connection — subsequent answers stream in ~2 s |
| Tests fail with "database not reachable" | Run tests via `docker compose exec` (they need the compose DB), not on the host |
| Start completely fresh | `docker compose down -v` (⚠ deletes the database **and** uploaded files) → `docker compose up --build` → re-seed |

## 13. Security

- **Secrets** live only in `.env` (gitignored; `.env.example` ships placeholders).
- **Passwords** bcrypt-hashed at rest; the seed is an upsert, so rotating
  `ADMIN_PASSWORD` in `.env` takes effect on the next boot.
- **JWT** with expiry on every admin route; missing/garbage/expired tokens → 401.
- **Uploads** validated by type (PDF / PNG / JPEG / WebP) and size — an early
  Content-Length reject plus an exact streaming cap that cleans up partial files.
- **SQL** is parameterized throughout — no string-built queries anywhere.
- **Rate limits** on the public chat (20/min/IP) and login (10/min/IP); question length
  capped (422).
- **CORS** restricted to the frontend origin.
- **Storage exposure minimized**: only reference images are publicly served; raw
  uploaded PDFs are not reachable.

## 14. Cost & performance strategy

| Lever | Effect |
|---|---|
| Vision at upload, not per query | The one multimodal cost is paid once per image; cost scales with uploads, not traffic |
| Lean hot path | Every question = 1 embedding + 1 hybrid SQL retrieval + 1 `gpt-4o-mini` call |
| Cheap models | `text-embedding-3-small` + `gpt-4o-mini` are fast and inexpensive, sufficient for grounded synthesis |
| Single store | Postgres + pgvector co-locates vectors, metadata, full-text, and analytics — no extra services |
| Hybrid without rerank | The lexical half is free in Postgres and rescues part numbers / error codes; a cross-encoder is consciously deferred |
| SSE streaming | First-token latency drops sharply — answers feel instant (~2 s typical full answer) |
| Abuse caps | Length caps and per-IP rate limits keep the public endpoint's worst-case spend bounded |

## 15. Consciously deferred

Recorded as decisions, not omissions:

- **Cross-encoder reranker** — adds per-query latency/cost; unjustified at this corpus size.
- **Query-time vision** — the demonstrated fix for lossy captions is a richer ingestion
  prompt (paid once); vision stays off the hot path.
- **True BM25** — Postgres `ts_rank_cd` is the pragmatic lexical signal; a `pg_search`
  extension could slot in without architecture changes.
- **Multi-image answers, session tracking, feedback capture (👍/👎)** — analytics loop v2.
- **Error-reason column for failed ingestions** — `docker compose logs backend` covers it.
- **MIME magic-byte sniffing** — uploads are admin-only and garbage self-heals to `FAILED`.
- **Response `max_tokens` cap** — grounded answers are naturally short.
